Style Guide

About the Style Guide

This Style Guide provides guidelines, code, and best practices for implementing a consistent user interface across the AGL open application suite.

Sections

The guide is divided into sections based on application components, HTML elements, and design themes. The following sections are currently available:

    How to use this Guide

    Guidelines

    Each section provides an overview of the how to use the components in your applications. This includes design rationale, contextual application, design variations, and detailed implementation instructions where needed.

    Additionally, the guide provides sections on writing semantic, quality code to improve collaboration.

    Code Snippets

    Code snippets are provided with most examples. The HTML snippets can be copied and used as a starting point for your application. The CSS snippets are for reference only as the styles will be integrated into common.css. *

    * CSS that has not been integrated into common.css will be noted as such.

    Most code snippets are contenteditable making it easy to select and copy code. However, this means you can inadvertantly edit the snippet. If you do, refreshing the browser will reset any edited snippets.

    Contributing to the Style Guide

    The Style Guide is a living document that will evolve as we enhance the application suite. If an addition or edit needs to be made, please use the following guidelines:

    • Collaborate on new design options with the Graphics Development team to see how new introductions will affect existing applications and benefit the overall user experience.
    • Consider whether a new component needs to be a common component. New components will only be added to common when they are shared by more than one application.
    • Provide sketches, mockups and/or code for new components.
    • Get code and design reviews for new components you're working on.
    • Develop new components in your applications or local branches.
    • Submit a Trac ticket for new component candidates or fixes/changes.
    • Contributions will be evaluated and planned for each sprint.
